# req4 Review Notes

日期：2026-03-25

本文汇总今天对 `reqs/req4.md` 对应改动的 review 过程中发现过的所有问题，不区分这些问题后来是否已经修复。

## 1. 验证脚本的关键 probe 点落在散热片实体内部

最早版本的 `validate_transient.py` 把 `(0.0, 0.0)` 设为关键 probe 点，用它来判断速度和温度的瞬态趋势。

问题在于，这个点落在中间那根 fin 的几何范围内，不属于流体域，而属于散热片实体区域。PINN 在这里并没有受到流体 PDE 和对应边界条件的正确约束，因此该点上的 `u/v/p/c` 输出不具备可靠的物理意义。

这会导致：

- “t=0 速度接近 0”
- “t=t_max 速度建立起来”
- “速度随时间增长”

这类趋势检查本身失真，验证结论不可信。

## 2. 验证脚本不能可靠重建训练时的真实网络结构

最早版本的 `validate_transient.py` 没有通过训练产物中的 `.hydra/config.yaml` 来恢复真实网络结构，而是直接尝试用默认 `FullyConnectedArch(...)` 加载 checkpoint。

如果训练时的 `fully_connected` 配置和默认值不完全一致，例如：

- 层数
- hidden size
- 激活函数
- weight norm
- 其他 arch 配置

那么验证脚本就可能：

- 直接加载失败
- 或者 fallback 到一个并不等价的伪网络结构

该 fallback 逻辑只是按 state dict key 排序后手工拼接 `Linear + SiLU`，会丢失原始架构语义，而且字符串排序还可能引入层顺序错误。这样即使图和误差数值能跑出来，也未必对应真实训练模型，验证结果可能是假的。

## 3. 瞬态问题在 `t=0` 存在“冲击启动”式初始/边界不相容

最早版本的 `heat_sink_transient.py` 中：

- 入口速度边界在整个 `t in [0, t_max]` 上直接施加满幅抛物线流速
- 热壁温度边界在整个 `t in [0, t_max]` 上直接施加固定高温
- 同时又把 `t=0` 的全域初始条件设为静止流体、环境温度

这相当于：

- `t=0` 瞬间启动风
- `t=0` 瞬间开启加热

这不是绝对错误的物理建模，但会让 `t=0` 成为一个很尖锐的突变点。对于 PINN 来说，这意味着时间起点附近更难学习，更容易出现：

- 收敛变慢
- 残差偏大
- 瞬态早期预测不稳定

本质上，这是一个“算法可行，但训练难度被不必要放大”的问题。

## 4. `1d_wave_equation` 示例的处理思路与当前瞬态热流问题不完全一致

在 review 过程中对照了 NVIDIA PhysicsNeMo 的 `1d_wave_equation` 文档。该示例的核心做法是：

- 显式引入时间变量
- 单独施加初始条件
- 用与初始状态自洽的方式定义瞬态起点

而它并没有像当前热流问题这样，在 `t=0` 同时引入“系统初始静止”与“边界激励瞬间打满”的阶跃外部驱动。

因此，如果直接拿这个示例作为“当前实现没有问题”的依据，是不充分的。两者在时间起点是否平滑这一点上有明显差异。

## 5. `t_max = 10` 的合理性最初缺少更明确的物理解释

在 review 过程中讨论过 `t in [0, 10]` 是否合理。仅从实现角度把时间区间写成 `0~10` 还不够，需要至少从物理量级上说明：

- 通道尺度上的对流特征时间大约是多少
- `t_max=10` 相当于多少个流动穿越时间
- 该时间窗是否足以接近稳态

对当前问题做数量级估算后，`t_max=10` 大致属于合理候选区间，但这件事在代码和说明中最初并没有明确表达，因此会让时间窗口看起来偏经验化、缺少物理依据。

## 6. 验证脚本的判据阈值偏经验化

`validate_transient.py` 中一些检查使用了经验阈值，例如：

- `t=0` 速度阈值
- `t=0` 温度阈值
- `t=t_max` 某点速度阈值

这些阈值可以作为工程上的 quick check，但如果没有说明其来源，就会带来两个问题：

- 检查通过不代表模型一定物理正确
- 检查失败也不一定说明模型真的错误

因此这类阈值更适合作为辅助体检项，而不是严肃的物理验收标准。

## 7. transient 验证链路没有接入主入口，交付流程不闭环

虽然仓库中新增了 `validate_transient.py`，但 review 过程中发现主入口仍然主要是：

- `run_case.sh`
- `run_case.py`

而它们没有自动调用 transient 专项验证，也没有自动把：

- 瞬态验证图片
- `transient_validation_report.md`

归档到本次 `outputs/transient/...` 的结果目录中。

这带来的问题是：

- 训练可以跑完
- 通用 report 可以生成
- 但 `req4.md` 要求的“测试脚本有图片和报告”没有真正接到主流程里

从软件架构和交付一致性的角度看，这仍然是一个问题。

## 8. transient 模式下仍沿用通用 baseline 报告语义

review 过程中还注意到，主流程输出的报告仍然主要是通用 `baseline_report.md` 风格。这会带来语义上的偏差：

- transient 有自己的验证目标
- transient 有自己的时间维度和专门检查
- 但主输出报告仍然是 baseline 风格

这会削弱最终产物的可读性，也不利于区分“训练监控报告”和“瞬态物理验证报告”。

## 总结

今天 review 过程中发现过的问题，主要集中在三类：

- 瞬态建模在 `t=0` 的物理一致性
- 验证脚本本身的可信度
- 主入口与验证交付链路是否闭环

其中最关键的技术问题是：

1. probe 点落在实体内部导致验证失真
2. 验证脚本不能可靠恢复真实网络结构
3. 初始条件与边界条件在 `t=0` 存在冲击式不相容

其余问题主要属于验证标准和软件交付流程上的完整性问题。
